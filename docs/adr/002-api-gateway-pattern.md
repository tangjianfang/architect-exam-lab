# ADR-002：采用 API Gateway 模式

| 字段 | 内容 |
|------|------|
| **ADR 编号** | ADR-002 |
| **标题** | 采用 API Gateway 作为系统统一入口 |
| **日期** | 2025-01-02 |
| **状态** | ✅ 已接受（Accepted）|
| **关联 ADR** | ADR-001（微服务架构）|
| **关联考点** | API Gateway 模式、横切关注点、边缘服务 |

---

## 背景与问题

基于 ADR-001 确定采用微服务架构后，客户端需要与多个微服务交互，这带来以下问题：

1. **多端点问题**：客户端需要知道所有服务地址（auth:3001, registry:3002, tool:3003...）
2. **认证分散**：每个服务都需要实现 JWT 验证，代码重复
3. **限流困难**：无法在客户端层面统一控制请求频率
4. **协议不统一**：未来部分服务可能使用 gRPC，需要协议转换
5. **日志分散**：请求日志散落在各服务，难以全局追踪

---

## 可选方案

### 方案一：直接访问（No Gateway）

客户端直接与各微服务通信。

**优点：**
- 无额外中间层，延迟最低
- 实现最简单

**缺点：**
- 客户端需要管理多个服务地址，耦合度高
- 认证、限流等横切关注点需在每个服务重复实现
- 无法统一日志和监控
- 暴露内部服务拓扑，安全风险

### 方案二：自建 API Gateway（Express.js + http-proxy）

使用 Express.js 实现自定义 API Gateway。

**优点：**
- 完全可控，可定制所有逻辑
- 学习价值高（适合备考实战）
- 轻量，无额外依赖
- 可精确控制各中间件的实现

**缺点：**
- 需要自行实现路由、认证、限流、熔断等功能
- 维护成本较高

### 方案三：使用 Kong / AWS API Gateway 等成熟产品

使用生产级的 API Gateway 产品。

**优点：**
- 功能丰富，开箱即用
- 生产验证，高可靠性
- 插件生态丰富

**缺点：**
- 引入外部依赖，增加学习曲线
- 备考角度：屏蔽了底层实现，减少考点覆盖
- 本地开发环境更复杂

---

## 决策结果

**选择方案二：自建 API Gateway（Express.js）**

**选择理由：**
1. **备考目标**：自建 Gateway 让我们亲自实现令牌桶限流、熔断器等算法，直接覆盖考点
2. **学习价值**：理解 API Gateway 的核心功能，对架构设计考试有直接帮助
3. **可控性**：项目初期规模不需要 Kong 这类重量级产品
4. **契合 Express 技术栈**：与其他服务技术栈一致，降低学习成本

## API Gateway 核心功能设计

```
客户端请求
    │
    ▼
┌──────────────────────────────────────────────────────────┐
│                    API Gateway 中间件管道                   │
│                                                          │
│  1. 日志记录中间件（请求开始）                               │
│  2. CORS 中间件                                          │
│  3. 请求体解析中间件（JSON/URLEncoded）                     │
│  4. 认证中间件（验证 JWT Token）                           │
│  5. 限流中间件（令牌桶算法，按 IP + 用户 ID）                │
│  6. 熔断器中间件（检查目标服务健康状态）                      │
│  7. 路由中间件（代理到目标微服务）                           │
│  8. 错误处理中间件（统一错误响应格式）                        │
│  9. 日志记录中间件（请求结束，记录耗时）                      │
└──────────────────────────────────────────────────────────┘
    │
    ▼ 代理到对应的微服务
```

### 路由规则

| 路径前缀 | 目标服务 | 认证要求 |
|---------|---------|---------|
| `POST /auth/login` | auth-service:3001 | 无需认证 |
| `POST /auth/register` | auth-service:3001 | 无需认证 |
| `POST /auth/refresh` | auth-service:3001 | 无需认证 |
| `GET /tools/*` | tool-registry:3002 | 需要认证 |
| `GET /health` | gateway 本身 | 无需认证 |
| `GET /metrics` | gateway 本身 | 需要管理员权限 |

## ATAM 质量属性影响分析

| 质量属性 | 影响方向 | 分析 |
|---------|---------|------|
| **性能（Performance）** | ⚠️ 轻微负面 | 增加一跳网络调用（约 1-5ms），通过连接复用和缓存补偿 |
| **可用性（Availability）** | ⚠️ 风险点 | Gateway 是单点，需要多副本部署（K8s Deployment replicas≥2） |
| **安全性（Security）** | ✅ 正面 | 统一认证入口，内部服务无需暴露到外网，减少攻击面 |
| **可修改性（Modifiability）** | ✅ 正面 | 横切关注点集中管理，修改一处即可影响所有服务 |
| **可测试性（Testability）** | ✅ 正面 | 中间件可独立单元测试，逻辑清晰 |
| **可观测性（Observability）** | ✅ 正面 | 统一日志格式，全链路请求追踪 |

### 权衡点

**网关单点 vs 多副本部署**
- Gateway 是系统的唯一入口，成为关键单点（SPOF）
- 缓解措施：K8s 部署 ≥2 个副本，配合 Readiness Probe 实现零停机更新

**横切关注点集中 vs 分散**
- 集中（当前方案）：改一处生效所有服务，但 Gateway 代码可能膨胀
- 分散（Service Mesh）：Sidecar 模式，每个服务独立处理，更灵活但复杂
