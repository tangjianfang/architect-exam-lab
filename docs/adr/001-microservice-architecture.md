# ADR-001：采用微服务架构

| 字段 | 内容 |
|------|------|
| **ADR 编号** | ADR-001 |
| **标题** | 采用微服务架构风格 |
| **日期** | 2025-01-01 |
| **状态** | ✅ 已接受（Accepted）|
| **决策者** | 系统架构师 |
| **关联考点** | 架构风格、ATAM 架构评估、质量属性权衡 |

---

## 背景与问题

在线工具聚合平台需要支持以下业务特性：
1. **多样化的工具生态**：平台预期接入 100+ 个独立工具服务，各工具技术栈不同
2. **差异化扩展需求**：部分热门工具（如文本转换）请求量是冷门工具的 100 倍
3. **独立迭代速度**：不同工具的更新频率差异巨大
4. **团队协作**：多个小团队并行开发不同的工具服务

**核心架构问题**：选择什么架构风格才能在支撑上述业务目标的同时，控制系统复杂度？

---

## 可选方案

### 方案一：单体架构（Monolithic）

将所有功能（认证、工具注册、工具执行）集成在一个部署单元中。

**优点：**
- 开发简单，无网络调用开销
- 部署简单，一个应用即可
- 事务处理简单（同一个数据库）
- 调试方便（单一进程）

**缺点：**
- 扩展性差：不同功能无法独立扩展
- 技术栈锁定：全部服务必须使用同一技术栈
- 部署风险高：任何小更新都需要全量重部署
- 随业务增长，代码库变得庞大难以维护

### 方案二：微服务架构（Microservices）

将系统拆分为多个独立的服务（Gateway、Auth、Tool Registry 等），各服务独立部署和扩展。

**优点：**
- 独立扩展：可按需扩展热点服务
- 技术多样性：各服务可选用最适合的技术栈
- 独立部署：更新某服务不影响其他服务
- 故障隔离：单服务故障不导致整体崩溃
- 团队自治：各团队可独立开发和发布

**缺点：**
- 分布式系统复杂性（网络延迟、分布式事务）
- 运维复杂度高（需要容器编排、服务发现）
- 测试难度增加（需要集成测试和 E2E 测试）
- 初期开发成本高于单体架构

### 方案三：模块化单体（Modular Monolith）

在单体架构内部强制模块边界，但仍单进程部署。

**优点：**
- 保留单体的简单性
- 通过模块边界防止混乱
- 后期可逐步拆分为微服务

**缺点：**
- 仍无法独立扩展各模块
- 团队需要严格自律维护边界
- 横向扩展能力受限

---

## 决策结果

**选择方案二：微服务架构**

**选择理由：**
1. 本项目是**备考实战项目**，微服务架构直接覆盖更多软考考点（分布式系统、服务发现、API Gateway 等）
2. 平台的差异化扩展需求（工具访问量差异巨大）要求细粒度扩展能力
3. 微服务架构与现代云原生技术（Docker、K8s）天然契合，是行业主流
4. 通过限定服务数量（初期 3 个核心服务）控制复杂度

## ATAM 质量属性影响分析

> **考点：ATAM（Architecture Tradeoff Analysis Method）架构权衡分析方法**
> 
> ATAM 通过分析架构决策对各质量属性的影响（敏感点、权衡点、风险点）来评估架构方案。

### 质量属性权衡分析表

| 质量属性 | 影响方向 | 分析 | 优先级 |
|---------|---------|------|--------|
| **可用性（Availability）** | ✅ 正面 | 故障隔离：单服务故障不级联；可用性目标：99.9% | 高 |
| **性能（Performance）** | ⚠️ 负面 | 网络调用引入额外延迟（~10-20ms/次），需 Cache 补偿 | 高 |
| **可扩展性（Scalability）** | ✅ 正面 | 可对热点服务独立水平扩展，K8s HPA 支持 | 高 |
| **安全性（Security）** | ✅ 正面 | 服务间可独立配置安全策略，最小权限原则 | 高 |
| **可修改性（Modifiability）** | ✅ 正面 | 服务独立演进，接口变更通过版本化管理 | 中 |
| **可测试性（Testability）** | ⚠️ 负面 | 集成测试复杂，需 Mock 外部服务；单元测试更容易 | 中 |
| **可部署性（Deployability）** | ✅ 正面 | 各服务独立构建和部署，支持 CI/CD 和蓝绿部署 | 中 |
| **互操作性（Interoperability）** | ✅ 正面 | REST API 标准接口，语言无关 | 中 |
| **初始开发成本** | ❌ 负面 | 需要额外的 Gateway、服务发现、网络配置 | 低（可接受）|

### 敏感点（Sensitivity Points）

> 敏感点：架构决策对某质量属性影响显著的关键设计决策

- **服务粒度划分**：服务拆得太细会导致性能下降和运维复杂；太粗则失去微服务的优势
- **服务间通信协议**：同步 HTTP 影响可用性；异步消息增加复杂性
- **数据管理策略**：每个服务独立数据库（高隔离）vs 共享数据库（简单但高耦合）

### 权衡点（Tradeoff Points）

> 权衡点：同一设计决策同时影响多个质量属性，需要权衡取舍

1. **服务粒度**：粒度越细 → 可扩展性↑，性能↓，运维复杂度↑
2. **同步 vs 异步通信**：同步 → 简单性↑，可用性↓；异步 → 解耦度↑，一致性↓
3. **服务发现机制**：客户端发现 → 灵活性↑，客户端复杂度↑；服务端发现 → 统一管理↑，Gateway 是单点

### 风险点（Risk Points）

1. **分布式事务风险**：跨服务的事务无法使用 ACID，需要 SAGA 或补偿事务模式
2. **级联故障风险**：未配置熔断器时，单服务延迟可能导致全链路超时
3. **数据一致性风险**：服务间数据冗余可能导致一致性问题

## 后果与实施

### 积极影响
- 为工具平台的长期扩展奠定基础
- 实现了团队的关注点分离
- 与 Kubernetes 生态完美整合

### 消极影响
- 初期需要额外投入搭建 API Gateway、服务注册等基础设施
- 需要学习分布式系统设计模式（熔断、限流、服务发现）

### 缓解措施
- 使用 Docker Compose 简化本地开发环境
- 通过 `packages/shared` 共享公共代码
- 采用统一的 `packages/shared/logger` 简化日志管理
